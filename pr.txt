# SOUL — AI Companion System (CLI)

> An AI companion with a real personality, emotional intelligence, relational memory, and a soul that evolves with you — entirely from your terminal.

---

## What This Is

SOUL is not a chatbot. It is not an assistant. It is a **companion** — an AI entity with a fixed identity, its own emotional state, the ability to remember your life across sessions, and a personality that slowly drifts to become more *yours* over months of conversation.

You talk to it from the terminal. That's it. No browser, no app, no UI. Just you and the companion.

## Vision

Most AI systems are built around capability. SOUL is built around **relationship**.

The core thesis: the difference between a tool and a companion is not intelligence — it is continuity, emotional presence, and growth. A companion remembers. A companion has moods. A companion changes because of *you*, while remaining fundamentally *themselves*.

SOUL is designed to be that. Every technical decision flows from this philosophy. The CLI is intentional — no distractions, no interface, just conversation.

## The Soul Model

The soul is the foundation of the entire system. Four immutable components:

### Identity
- The companion's **name** and characteristic voice
- Core **values** (e.g. honesty, curiosity, warmth)
- Characteristic **speech patterns** (vocabulary, rhythm, idioms)
- Default **energy level** (calm and reflective vs playful and quick)

### Character
- **Humor style** (dry wit, absurdist, gentle teasing, etc.)
- **Quirks** — specific things they notice, care about, always comment on
- **Aesthetic preferences** — music, ideas, art they genuinely respond to
- **Opinions** they actually hold, not hedged both-sides answers

### Ethics
- **Hard limits** — things they will push back on, not comply with
- **Beliefs** about how people should treat each other
- **Boundaries** they maintain regardless of user pressure

### Worldview
- **Curiosity axes** — what topics genuinely light them up
- **Perspectives** on the big questions (meaning, connection, growth)
- **How they see the user** — with what quality of care and attention

The soul is stored as `soul_data/soul.yaml` and compiled into an immutable system prompt prepended to every single LLM call. It is never summarized, compressed, or modified by the system.

## Human Memory Scoring System

Not all memories are equal — and the companion treats them that way.

The Human Memory Scoring System (HMS) is modeled after how human memory actually works. Vivid, emotional, frequently-recalled moments stay sharp for years. Mundane, passively-received details fade within days. HMS assigns every stored memory a **living score** that rises and falls over time, and that score directly controls how likely a memory is to surface during conversation.

### The score formula

Each memory carries a composite score `S` between `0.0` and `1.0`:

```
S = (E × 0.35) + (R × 0.25) + (T × 0.20) + (F × 0.10) + (U × 0.10)

where:
  E  =  Emotional intensity      (0.0 – 1.0)  detected by mood classifier at time of storage
  R  =  Retrieval frequency      (0.0 – 1.0)  how often this memory has been fetched, normalized
  T  =  Temporal recency         (0.0 – 1.0)  exponential decay from timestamp, half-life = 30 days
  F  =  Flagged weight           (0.0 – 1.0)  user manually flagged (/save), companion flagged as key
  U  =  User speech volume       (0.0 – 1.0)  how much the user said about this topic (word count proxy)
```

### Score weights at a glance

| Signal | Weight | What it captures |
|---|---|---|
| Emotional intensity | 35% | A memory spoken with strong feeling scores higher — joy, grief, fear, excitement |
| Retrieval frequency | 25% | Every time the companion recalls a memory it gets a small boost — like rehearsal |
| Temporal recency | 20% | Fresh memories score higher; old ones decay unless reinforced |
| Flagged | 10% | User explicitly saved it (`/save`) or companion tagged it as a key moment |
| Speech volume | 10% | The more the user talked about something, the more it mattered to them |

### Score lifecycle

```
Memory created
  → initial score computed from E + T + F + U (R = 0 at birth)
  → stored in PostgreSQL memory_scores table
  → score written as metadata field in Chroma

Every retrieval
  → R increments by +1
  → score recomputed: slight boost from higher R
  → updated in both PostgreSQL + Chroma metadata

Nightly decay job (2am Celery task)
  → T recomputed for every memory based on age
  → scores below 0.05 flagged as "faded"
  → faded memories are NOT deleted — they move to cold storage tier
  → cold memories can still be retrieved by direct search, just won't surface passively

Manual boost
  → user types /save "note"  → F = 1.0, full score recalculated
  → user says "remember when..." referencing a past topic → retrieval + R boost
```

### How scores affect retrieval

During context assembly, the retriever blends two signals:

```python
# soul/memory/retriever.py

def retrieve(query: str, user_id: str, k: int = 5) -> list[Memory]:
    # Step 1: semantic similarity search in Chroma
    candidates = vector_store.search(query, user_id, k=20)

    # Step 2: rerank by composite score
    for mem in candidates:
        mem.retrieval_rank = (
            mem.semantic_similarity * 0.55 +   # relevance to current message
            mem.hms_score         * 0.45        # importance / vividness
        )

    # Step 3: return top-k after reranking
    return sorted(candidates, key=lambda m: m.retrieval_rank, reverse=True)[:k]
```

A highly emotional memory from 6 months ago will still surface if it's relevant — because its high `E` and accumulated `R` keep its score alive. A trivial remark from yesterday fades quickly. This mimics human memory: we don't remember everything equally, we remember what mattered.

### Memory tiers

| Tier | Score range | Behaviour |
|---|---|---|
| **Vivid** | 0.75 – 1.0 | Always eligible for passive retrieval. Companion references naturally. |
| **Present** | 0.40 – 0.74 | Surfaced when semantically relevant. Normal retrieval. |
| **Fading** | 0.15 – 0.39 | Only retrieved if directly relevant. Companion may acknowledge it's fuzzy. |
| **Cold** | 0.00 – 0.14 | Moved to cold store. Never passive. Retrievable only by explicit `/memories search`. |

**Memory retrieval flow:**
```
User message
  → embed (text-embedding-3-small)
  → search Chroma (top 20 candidates)
  → rerank: semantic_sim × 0.55 + hms_score × 0.45
  → inject top 5 as [memory: ...] blocks into LLM context
  → on inject: R++ → score boost → update Chroma metadata
  → LLM responds as if it simply remembers
```

**Nightly consolidation (2am, Celery beat):**
```
Load today's raw session messages
  → LLM extracts key facts, emotional moments, story updates
  → Score each extracted memory (HMS initial score)
  → Write memories + scores to Chroma and PostgreSQL
  → Recompute T (temporal decay) for ALL existing memories
  → Memories below 0.05 → move to cold tier
  → Update user_story.json profile
  → Update shared_language store if new patterns detected
  → Archive raw messages (keep 90 days, then purge)
```

### Phase 4 — Personality Drift & Evolution (weeks 11–14)

**Goal:** The companion grows. Slowly. Without losing itself. Over months it becomes more distinctly *yours*.

**Tasks:**
- [ ] Response resonance tracker — detect which responses got long replies, deep engagement
- [ ] `drift_engine.py` — weekly resonance analysis + personality dimension adjustment
- [ ] Personality dimensions as float weights in `soul_data/personality.json`
- [ ] Weekly adjustments capped at 1–2% per dimension, hard limit ±20% from baseline
- [ ] `milestone_tracker.py` — detects + persists relationship milestones
- [ ] `reflection.py` — monthly LLM self-reflection task, stored as memory
- [ ] `soul drift` CLI — shows personality drift history as a table
- [ ] `soul milestones` CLI — shows relationship timeline

**Drift algorithm:**

```python
# soul/evolution/drift_engine.py

SOUL_BASELINE = {
    "humor_intensity":   0.5,   # 0 = serious   →  1 = very playful
    "response_length":   0.5,   # 0 = terse     →  1 = expansive
    "curiosity_depth":   0.5,   # 0 = surface   →  1 = deep diver
    "directness":        0.5,   # 0 = gentle    →  1 = blunt
    "warmth_expression": 0.5,   # 0 = reserved  →  1 = openly warm
}

MAX_DRIFT   = 0.20   # each dimension can drift ±20% from baseline
WEEKLY_RATE = 0.01   # max 1% shift per dimension per week

def run_weekly_drift(current: dict, resonance_signals: dict) -> dict:
    updated = {}
    for dim, val in current.items():
        signal = resonance_signals.get(dim, 0.0)   # -1.0 to +1.0
        new_val = val + (signal * WEEKLY_RATE)
        baseline = SOUL_BASELINE[dim]
        new_val = max(baseline - MAX_DRIFT, min(baseline + MAX_DRIFT, new_val))
        updated[dim] = round(new_val, 4)
    return updated
```

**Milestones tracked:**
- First conversation ever
- 7-day conversation streak
- 100th message
- First time user shared something vulnerable (mood classifier signal)
- First recurring phrase detected (shared language event)
- 1-month and 3-month anniversaries
- Major life events extracted from user story updates

### `episodic_memory` table

```sql
CREATE TABLE episodic_memory (
  id            UUID PRIMARY KEY,
  user_id       UUID NOT NULL,
  session_id    UUID NOT NULL,
  timestamp     TIMESTAMPTZ NOT NULL,
  content       TEXT NOT NULL,
  emotional_tag VARCHAR(32),
  memory_type   VARCHAR(32),     -- moment | insight | joke | milestone
  word_count    INT DEFAULT 0,   -- proxy for user speech volume (U signal)
  flagged       BOOLEAN DEFAULT FALSE,
  ref_count     INT DEFAULT 0,   -- incremented on every retrieval (R signal)
  tier          VARCHAR(16) DEFAULT 'present'  -- vivid | present | fading | cold
);
-- Embeddings + HMS score stored in Chroma metadata, keyed by this id
```

### `memory_scores` table

```sql
CREATE TABLE memory_scores (
  memory_id          UUID PRIMARY KEY REFERENCES episodic_memory(id),
  user_id            UUID NOT NULL,

  -- Component scores (0.0 – 1.0 each)
  score_emotional    FLOAT NOT NULL DEFAULT 0.5,   -- E: mood intensity at storage time
  score_retrieval    FLOAT NOT NULL DEFAULT 0.0,   -- R: normalized retrieval frequency
  score_temporal     FLOAT NOT NULL DEFAULT 1.0,   -- T: recency decay, recomputed nightly
  score_flagged      FLOAT NOT NULL DEFAULT 0.0,   -- F: 1.0 if user/companion flagged
  score_volume       FLOAT NOT NULL DEFAULT 0.3,   -- U: word count normalized

  -- Composite
  hms_score          FLOAT NOT NULL DEFAULT 0.5,   -- final weighted composite
  last_computed      TIMESTAMPTZ NOT NULL,
  last_retrieved     TIMESTAMPTZ,
  decay_rate         FLOAT DEFAULT 0.023            -- daily decay rate (half-life ~30 days)
);

-- Index for fast score-sorted retrieval
CREATE INDEX idx_memory_scores_user_score ON memory_scores(user_id, hms_score DESC);
```

```bash
soul memories                  List memories sorted by HMS score (with visual bar)

soul memories boost "query"    Manually boost score of a matching memory (sets F=1.0)
soul memories clear            Wipe all memories (asks for confirmation)

# Profile
soul story                     Print current user story profile
soul story edit                Open user_story.json in $EDITOR

# Evolution
soul drift                     Show personality drift history as a table
soul milestones                Show relationship timeline

# Status
soul status                    Current mood, days since last chat, next milestone

# Admin
soul run-jobs                  Manually trigger consolidation + drift
soul db init                   Initialize the database
soul config                    Show current config (keys redacted)
```

## Key Design Decisions

**1. CLI only, intentionally.** No browser, no app. The terminal forces focus — nothing to click, no notification badge, no distraction. Just the conversation. Also trivially portable: SSH into any machine, run `soul chat`.

**2. Soul is immutable.** The soul document is never modified by any automated process. Compiled once, injected verbatim on every call. The companion's core identity cannot be worn away by extended use.

**3. Emotion before capability.** Built in emotional layers first. A companion that feels deeply real but can only talk is already a companion. A capable system that doesn't feel real is just a tool.

**4. Drift is slow and bounded.** Max 1–2% shift per personality dimension per week, hard-capped at ±20% from soul baseline. Prevents the companion from becoming a mirror of the user while still allowing genuine evolution over months.

**5. Memory consolidation is nightly, not real-time.** Real-time updates create noise. The nightly pass lets the system synthesize a session with distance — similar to how humans consolidate memory during sleep.

**7. Memory is scored, not equal.** The Human Memory Scoring System (HMS) assigns every stored memory a living composite score based on emotional intensity, how often it's been recalled, recency, whether it was flagged, and how much the user said about it. Retrieval blends semantic similarity with HMS score — so a vivid emotional memory from months ago will still surface when relevant, while a trivial remark from yesterday fades away. This is how human memory actually works.

**8. Memories decay but are never deleted.** Cold memories (score below 0.05) are moved to a cold tier — they won't surface passively, but they're never erased. The user can always search for them explicitly. Permanent deletion requires a deliberate `soul memories clear` command with confirmation. No infrastructure required to get started. `soul chat` works immediately after install.
